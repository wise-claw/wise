# 整合 Phase 3+ 路线图

## 背景

Phase 2 已落地基于别名的整合与 Tier-0 契约保护，覆盖：

- `ralplan`
- `team`
- `ralph`
- `ultrawork`
- `autopilot`

本路线图定义下一波工作：智能体利用率清理、路由简化与迁移治理。

## 目标

1. 在不破坏兼容性的前提下缩减智能体面。
2. 改善路由质量（合适的智能体、合适的层级、更少的空闲/重复委派）。
3. 将已弃用策略与发布安全门正式化。

## 范围

### 1) 智能体目录整合

- 建立规范泳道：
  - 发现
  - 规划/分析
  - 实现
  - 验证/审查
- 将 legacy/重叠角色标记为兼容别名。
- 为旧名保留稳定的兼容映射。

### 2) 路由与利用率

- 增加从技能家族到规范智能体泳道的显式路由矩阵。
- 增加遥测信号：
  - 调用次数
  - 完成率
  - 重试率
  - 升级率
- 定义 "保留 / 合并 / 弃用" 的阈值。

### 3) 迁移治理

- 层级分类：
  - Tier-0：不可变公开契约（已强制）
  - Tier-1：稳定核心
  - Tier-2：整合候选
- 非 Tier-0 名称最少两个发布版本的弃用窗口。
- 经路由清单开关提供回滚护栏。

## 验收标准

- 规范智能体矩阵已文档化并从 `docs/参考.md` 链接。
- 兼容别名为既有名称保持可用。
- 回归测试覆盖：
  - 别名保真
  - 受保护模式不变量
  - 文档/运行时一致性检查
- 无 Tier-0 行为回归。

## 拟定交付计划

### 里程碑 A — 发现 + 指标

- 盘点当前智能体使用情况与重叠。
- 基于证据提出保留/合并/弃用候选。

### 里程碑 B — 运行时路由清理

- 实现路由表变更 + 兼容别名。
- 增加针对智能体解析行为的定向测试。

### 里程碑 C — 文档 + 迁移策略

- 发布弃用时间表与迁移说明。
- 更新 AGENTS/文档一致性检查。

### 里程碑 D — 验证门

- 运行完整验证：
  - `npm test`
  - `npm run build`
  - `npm run lint`
- 验证无 Tier-0 回归。

## 风险

- 过度裁剪专业化智能体会降低边缘任务质量。
- hooks 与特定智能体名之间存在隐藏耦合。
- 命名变更未同步时文档漂移。

## 风险控制

- 别名优先迁移（绝不先硬删除）。
- 每个整合 PR 都要求受保护模式回归套件。
- 渐进式发布，并有清晰回滚路径。
