<a id="installation"></a>
## 安装

**仅支持 Claude Code 插件安装方式。** 其他安装方式（npm、bun、curl）已弃用，可能无法正常工作。

### Claude Code 插件（必需）

```bash
# Step 1: Add the marketplace
/plugin marketplace add https://github.com/wise-claw/wise

# Step 2: Install the plugin
/plugin install wise
```

该方式直接集成 Claude Code 插件系统，并使用 Node.js hooks。

> **说明**：插件安装流程**不支持**直接 npm/bun 全局安装。若仅需打包 CLI 能力，npm 包同时暴露 `wise` 与 `wise` 两个命令；示例中优先使用 `wise`，仅在排查问题时才需要长别名。

### 前置条件

- 已安装 [Claude Code](https://docs.anthropic.com/claude-code)
- 满足以下之一：
  - **Claude Max/Pro 订阅**（个人用户推荐）
  - **Anthropic API key**（`ANTHROPIC_API_KEY` 环境变量）

<a id="configuration"></a>
## 配置

### 项目级配置（推荐）

仅为当前项目配置 wise：

```
/wise:wise-setup --local
```

- 在当前项目创建 `./.claude/CLAUDE.md`
- 配置仅作用于本项目
- 不影响其他项目或全局设置
- **安全**：保留你的全局 CLAUDE.md

### 全局配置

为所有 Claude Code 会话配置 wise：

```
/wise:wise-setup
```

- 全局创建 `~/.claude/CLAUDE.md`
- 配置作用于所有项目
- **默认**：显式覆盖已有 `~/.claude/CLAUDE.md`
- **可选保留模式**：保留基础文件，将 WISE 写入 `~/.claude/CLAUDE-wise.md`，并在启动时让 `wise` 强制加载该伴随配置，而纯 `claude` 保持不变

### 配置带来的能力

| 功能           | 未配置     | 配置 wise 后         |
| -------------- | ---------- | -------------------- |
| Agent 委派     | 仅手动     | 按任务自动委派       |
| 关键词检测     | 禁用       | ultrawork、search    |
| Todo 续跑      | 基础       | 强制完成             |
| 模型路由       | 默认       | 智能层级选择         |
| 技能组合       | 无         | 自动组合技能         |

### 配置优先级

若两种配置同时存在，**项目级优先于**全局：

```
./.claude/CLAUDE.md  (project)   →  Overrides  →  ~/.claude/CLAUDE.md  (global)
```

#### 使用 `WISE_STATE_DIR` 集中存储 state

默认情况下，WISE 将 state 存于 `{worktree}/.wise/`，worktree 删除后会丢失。若要在 worktree 生命周期内保留 state，请设置 `WISE_STATE_DIR`：

```bash
# In your shell profile (~/.bashrc, ~/.zshrc, etc.)
export WISE_STATE_DIR="$HOME/.claude/wise"
```

解析结果为 `~/.claude/wise/{project-identifier}/`，其中项目标识符使用 git remote URL 的哈希（跨 worktree/clone 稳定），纯本地仓库则回退为目录路径哈希。

若 legacy `{worktree}/.wise/` 与集中式目录同时存在，WISE 会记录提示并使用集中式目录。之后可将 legacy 目录数据迁移并删除。

<a id="multi-repo-workspaces-with-wise-workspace"></a>
#### 使用 `.wise-workspace` 的多仓库工作区

当多个独立 git 仓库位于同一父目录下，且父目录本身**不是** git 仓库时，WISE 无法通过 `git rev-parse --show-toplevel` 推断共享根。每个子仓库会得到各自隔离的 `.wise/`。若要在父目录锚定单一 `.wise/`，可在该处放置 `.wise-workspace` 标记文件：

```bash
cd /path/to/my-workspace            # parent dir (not a git repo)
echo '{}' > .wise-workspace          # empty JSON is fine
```

从任意子目录（含任一子 git 仓库内）运行时，WISE 都会将 `.wise/` 解析为 `/path/to/my-workspace/.wise/`。标记也可携带显式项目标识符，使所有会话共享 state，而不依赖父目录名：

```json
{ "id": "my-org-bidchex" }
```

`getWiseRoot()` 内解析顺序：

1. `WISE_STATE_DIR`（集中式）。
2. `.wise-workspace` 标记（多仓库工作区）。
3. `git rev-parse --show-toplevel`（monorepo / 单仓库）。
4. `process.cwd()`（最后手段）。

工作区锚定后，不同子仓库中的多个 Claude Code 会话可并行运行 `/ultragoal`、`/ralph`、`/ultrawork`、`/autopilot` 而不串 state。对 `/ultragoal`，请在 `create-goals` 上传递 `--plan-id <id>` 或 `--auto-plan-id`，使各会话写入 `.wise/ultragoal/plans/{planId}/` 而非共享 `goals.json` — 见下方「ultragoal 多计划」。`session-start.mjs` 中的 PARALLEL SESSION WARNING 会执行 PID 感知存活检查，owner 会话已结束时不再抑制 restore。

#### `.wise/handoffs/` 共享契约

`.wise/handoffs/` 按设计**在 team 运行间共享**。其用途是会话间消息传递：team 阶段 handoff（plan → prd → exec → verify）在此累积，后续 `team` 运行可从最后非终端阶段恢复而不丢失决策历史。

**仅 `team` 技能会写入 `.wise/handoffs/`。** 其他读取该目录的代码均为只读。由 lint 测试 `tests/lint/handoffs-writers.test.ts` 强制：`src/**` 与 `templates/**` 中，除 `src/team/` 或 `src/hooks/team-pipeline/` 外，任何将 `handoffs/` 作为写入目标的引用都会导致失败。

- Handoff 文件在 `TeamDelete` 与会话取消后**有意保留** — 它们是事后分析制品。
- 除非 `team` 技能明确演进为 per-session 收件箱（ADR 中跟踪的后续项），否则**不要**将 `.wise/handoffs/` session-scope 化。

#### 品牌路径类型（`ReadPath` / `WritePath`）

State 文件路径解析由 `src/lib/worktree-paths.ts` 中的 `resolveSessionStatePaths()` 返回带品牌标记的结构：

```ts
interface SessionStatePaths {
  sessionScoped: string;
  legacy: string;
  effectiveRead: ReadPath;   // string & { __brand: 'ReadPath' }
  effectiveWrite: WritePath; // string & { __brand: 'WritePath' }
}
```

该品牌可在编译期阻止 hook 将 read-fallback 路径静默传给 writer（或反之）— TypeScript 会拒绝交叉赋值。合法的品牌生产者仅有 `resolveSessionStatePaths()` 本身；`eslint.config.js` 中的 ESLint `no-restricted-syntax` 规则会阻止在 `worktree-paths.ts` 及其测试之外使用 `as ReadPath` / `as WritePath` 断言。编译期回归守卫位于 `src/lib/__tests__/session-state-paths.type-test.ts`。

<a id="legacy-state-migration-wise_migrate_legacy_state"></a>
#### Legacy state 迁移（`WISE_MIGRATE_LEGACY_STATE`）

在已有 `{worktree}/.wise/state/` 的仓库上启用 `WISE_STATE_DIR` 或 `.wise-workspace` 时，可 opt-in 一次性将 legacy state 复制到新的 session-scoped 路径：

```bash
export WISE_MIGRATE_LEGACY_STATE=1
```

语义：

- **触发**：由通过 migration helper 包装写入的 state 文件读取方，在每次读取时检查一次。
- **操作**：将 `{wiseRoot}/state/{name}-state.json` → `{wiseRoot}/state/sessions/{sessionId}/{name}-state.json`，使用原子 `.migrating` 哨兵 + rename 以支持崩溃恢复。
- **幂等**：session-scoped 文件已存在时，再次设置该 flag 为 no-op。
- **仅 opt-in**：永不自动触发；仅当 `WISE_MIGRATE_LEGACY_STATE=1` 时生效。
- **勿长期设置**：不要永久写入 shell profile；迁移会话设一次即可，之后 unset。

<a id="rollback--disable-multi-repo-wise_disable_multirepo"></a>
#### 回滚 / 禁用多仓库（`WISE_DISABLE_MULTIREPO`）

若 workspace 标记解析导致异常行为（例如删除过期的 `.wise-workspace` 标记后），可通过一个环境变量切换禁用多仓库路径解析：

```bash
export WISE_DISABLE_MULTIREPO=1
```

精确语义：

- **跳过** `.wise-workspace` 标记检测 — `findWorkspaceRoot()` 立即返回 `null`。
- **回退**为标准 `git rev-parse --show-toplevel` → `process.cwd()` 解析顺序。
- **保留**已设置的 `WISE_STATE_DIR` — 集中式 state 存储仍可用。
- **作用域**：按进程；在运行 `claude` 的 shell 会话中设置，而非项目级。

要恢复多仓库行为，unset 该变量：

```bash
unset WISE_DISABLE_MULTIREPO
```

#### Ultragoal 多计划布局

默认布局（单计划，monorepo / 单会话）：

```
.wise/ultragoal/brief.md
.wise/ultragoal/goals.json
.wise/ultragoal/ledger.jsonl
```

多计划布局，由 `wise ultragoal create-goals` 上的 `--plan-id <id>` 或 `--auto-plan-id` 启用：

```
.wise/ultragoal/plans/{planId}/brief.md
.wise/ultragoal/plans/{planId}/goals.json
.wise/ultragoal/plans/{planId}/ledger.jsonl
```

`--auto-plan-id` 从 brief 标题推导 `{epochMs}-{slug}`，因此两个并行会话运行 `wise ultragoal create-goals --auto-plan-id ...` 不会冲突。后续命令（`status`、`add-goal`、`complete-goals`、`checkpoint`、`record-review-blockers`）在仅有一个 plan 时自动解析；多个 plan 时需 `--plan-id <id>`。`wise ultragoal list-plans` 枚举可用 plan id。

### 何时重新运行 setup

- **首次**：安装后运行（选择项目或全局）
- **更新后**：重新运行以获取最新配置
- **不同机器**：在每台使用 Claude Code 的机器上运行
- **新项目**：在需要 wise 的每个项目中运行 `/wise:wise-setup --local`

> **注意**：更新插件后（通过 `npm update`、`git pull` 或 Claude Code 插件更新），**必须**重新运行 `/wise:wise-setup` 以应用最新 CLAUDE.md 变更。

### 远程 WISE / 远程 MCP 访问

Issue #1653 询问 WISE 是否能「连接远程 WISE」，使一台开发机浏览 lab/test 机器上的文件而无需交互式 SSH。

当前清晰、一致的答案是：

- **支持**：通过统一 MCP registry 连接**远程 MCP 服务器**
- **未实现**：通用「WISE 集群」、共享远程文件系统视图，或自动 remote-WISE 联邦
- **完整远程 shell 工作流仍适用**：SSH、worktree，或挂载/网络文件系统

若远程主机已暴露 MCP 端点，将其加入 MCP registry（或 Claude settings，然后重新运行 setup 以便 WISE 同步 registry 到 Codex）：

```json
{
  "mcpServers": {
    "remoteWise": {
      "url": "https://lab.example.com/mcp",
      "timeout": 30
    }
  }
}
```

这为 WISE 提供了连贯的远程 MCP 工具连接面。它**不会**让所有远程文件自动呈现为本地工作区，也**不会**替代 SSH 进行任意 shell 访问。

若未来需要更丰富的跨机器行为，需要单独的认证远程执行/文件系统设计，而非拉伸当前本地工作区架构。

### 通过 MCP 的公司上下文

WISE 还在现有 MCP 面上支持窄化的 company-context 契约。

在标准 WISE 配置文件中配置：

- 项目：`.claude/wise.jsonc`
- 用户：`~/.config/claude-wise/config.jsonc`

```jsonc
{
  "companyContext": {
    "tool": "mcp__vendor__get_company_context",
    "onError": "warn",
  },
}
```

- `tool` 为要调用的完整 MCP 工具名。
- `onError` 控制 prompt 级回退：`warn`、`silent` 或 `fail`。
- MCP 服务器仍通过常规 Claude/WISE MCP setup 路径注册。

这仍是 prompt 级工作流契约，而非运行时强制。完整接口、触发阶段与信任边界见 [company-context-interface.md](./company-context-interface.md)。

### Agent 自定义

编辑 `~/.claude/agents/` 中的 agent 文件以自定义行为：

```yaml
---
name: architect
description: Your custom description
tools: Read, Grep, Glob, Bash, Edit
model: opus # or sonnet, haiku
# Optional: effort inherits from the parent Claude Code session unless you add an explicit override.
# effort: high
---
Your custom system prompt here...
```

捆绑的 WISE agent prompt 当前**不**附带 `effort:` frontmatter 字段。`agents/*.md` 内的 effort 表述属于 prompt 正文的行为指引；除非 agent markdown 显式声明 override，否则运行时 effort 继承自父 Claude Code 会话。

### Stop 回调通知标签

使用 `wise config-stop-callback` 配置 Telegram/Discord stop 回调标签。

```bash
# Set/replace tags
wise config-stop-callback telegram --enable --token <bot_token> --chat <chat_id> --tag-list "@alice,bob"
wise config-stop-callback discord --enable --webhook <url> --tag-list "@here,123456789012345678,role:987654321098765432"

# Incremental updates
wise config-stop-callback telegram --add-tag charlie
wise config-stop-callback discord --remove-tag @here
wise config-stop-callback discord --clear-tags

# Inspect current callback config
wise config-stop-callback telegram --show
wise config-stop-callback discord --show
```

标签行为：

- Telegram：`alice` 规范化为 `@alice`
- Discord：支持 `@here`、`@everyone`、数字用户 ID（`<@id>`）及角色标签（`role:<id>` → `<@&id>`）
- `file` 回调忽略标签选项

<a id="runtime-storage-and-goal-artifacts"></a>
## 运行时存储与目标制品

WISE 文档应先按逻辑角色描述目标与工作流制品，再映射到运行时特定的存储根。不要将 `.omx/` 当作通用路径：它是 legacy OMX 运行时根，而 WISE 使用 `.wise/` 存放本地项目 state。

### 运行时根映射

| 运行时                        | 项目本地根             | 用户/全局根                   | 说明                                                                                                                                                     |
| ----------------------------- | ---------------------- | ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| WISE                          | `.wise/`               | `~/.wise/`                    | WISE 项目本地 state、plans、notepad、logs、research 与 ask 制品的规范存储根。                                                                            |
| OMX compatibility/runtime-v1  | `.omx/`                | `~/.omx/`                     | 旧版 OMX 会话与跨运行时 handoff 的兼容根。仅在文档化 OMX 特定行为时提及。                                                                                |
| OMO native                    | runtime-owned OMO path | runtime-owned OMO user path   | OMO 原生存储由该运行时拥有。WISE 文档应命名逻辑制品角色，除非 OMO 命令显式文档化具体路径。                                                               |

### 逻辑目标制品角色

编写文档或 handoff 时使用这些名称，使同一概念在 WISE、OMX 兼容与 OMO 原生运行时间可移植：

| 逻辑角色            | WISE 路径                                                                    | OMX 兼容路径                                                                 | 用途                                                                                                                                                                               |
| ------------------- | ---------------------------------------------------------------------------- | ---------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 目标/规格制品       | `.wise/specs/<slug>.md` 或 `.wise/plans/<slug>.md`                           | `.omx/specs/<slug>.md` 或 `.omx/plans/<slug>.md`                           | 用户目标、约束、验收标准与执行 handoff 的持久陈述。                                                                                                                                |
| 已批准执行计划      | `.wise/plans/<slug>.md`                                                      | `.omx/plans/<slug>.md`                                                       | 供 team 或 ralph 等执行工作流消费的已审计划。                                                                                                                                      |
| 任务/运行时 state   | `.wise/state/<mode>.json` 或 `.wise/state/sessions/<session-id>/<mode>.json` | `.omx/state/<mode>.json` 或 `.omx/state/sessions/<session-id>/<mode>.json` | 机器可读工作流 state。存在 session-scoped state 时优先于 legacy 扁平文件。                                                                                                       |
| Team 协调 state     | `.wise/state/team/<team-name>/...`                                           | `.omx/state/team/<team-name>/...`                                            | Worker 任务文件、mailbox、status、events 与 dispatch 元数据。worktree 支持的 worker 应使用 `WISE_TEAM_STATE_ROOT`/兼容 env 定位 leader 拥有的协调根。                            |
| Ask/advisor 制品    | `.wise/artifacts/ask/<provider>-<slug>-<timestamp>.md`                       | `.omx/artifacts/ask/<provider>-<slug>-<timestamp>.md`                        | 来自 `wise ask` 或兼容包装器的持久 advisor 输出。                                                                                                                                  |
| 计划级 notepad      | `.wise/notepads/<plan-name>/`                                                | `.omx/notepads/<plan-name>/`                                                 | 规划或执行命名目标时收集的持久笔记。                                                                                                                                               |
| 项目记忆            | `.wise/project-memory.json` 与 `.wise/notepad.md`                            | `.omx/project-memory.json` 与 `.omx/notepad.md`                              | 可复用项目事实与会话笔记。                                                                                                                                                         |

### `/goal` 互操作说明

<a id="plugin-directory-flags"></a>
## 插件目录标志

通过本地开发 checkout 而非 marketplace 插件启动 WISE 时，可配置 WISE 如何发现 agents、skills 与 commands。

> **本地开发推荐**：使用 `wise --plugin-dir <path>`（配合 `wise setup --plugin-dir-mode`）。与 `claude plugin marketplace add` 不同，该流程**无插件缓存**，直接从 checkout 加载 agents/skills，下次会话即可生效，无需 `marketplace update` / `plugin update` 往返 — 迭代更快。

### `wise --plugin-dir <path>`

**用法**：非消耗型启动 flag，捕获本地 checkout 路径。

```bash
wise --plugin-dir /path/to/wise setup --plugin-dir-mode
```

- **作用**：解析 `--plugin-dir <path>`（或 `--plugin-dir=<path>`）为绝对路径，设置 `WISE_PLUGIN_ROOT` 环境变量，然后将 flag 原样传给 Claude Code。
- **非消耗型**：flag 保留在参数列表中，Claude Code 插件加载器仍可见。
- **优先级**：显式 `--plugin-dir` flag 优先于已有 `WISE_PLUGIN_ROOT` env（不一致时警告）。
- **解析**：相对路径经 `path.resolve()` 转为绝对路径。注意：`~` **不会**展开 — 请用 `$HOME` 或绝对路径。
- **配合 setup**：仅 `--plugin-dir` 只影响当前 Claude 会话。还必须运行 `wise setup --plugin-dir-mode`（或让 `WISE_PLUGIN_ROOT` 触发自动检测），以便 HUD、hooks 与 CLAUDE.md 为链接的 checkout 安装。跳过此步会导致 `~/.claude/` 仍指向过期的插件根。

### `claude --plugin-dir <path>`（直接）

**用法**：不使用 `wise` shim 直接启动 Claude Code 时。

```bash
export WISE_PLUGIN_ROOT=/path/to/wise
claude --plugin-dir /path/to/wise
```

- **要求**：须手动设置 `WISE_PLUGIN_ROOT`，使 HUD wrapper 等 env 感知组件与插件加载器解析同一路径。
- **原因**：HUD bundle 需知 agents/skills/commands 位置，以与插件实例同步。
- **说明**：纯 `claude`（无 `wise`）不会自动为你捕获 `--plugin-dir`。

### `wise setup --plugin-dir-mode`

**用法**：setup 时显式启用 dev plugin-dir 模式。

```bash
wise setup --plugin-dir-mode
```

- **作用**：跳过将 agents 与捆绑 skills 复制到 `~/.claude/`，因插件已通过 `--plugin-dir` 在运行时提供。
- **仍会安装**：
  - HUD bundle（`~/.claude/hud/`）
  - Git hooks（`.git/hooks/`，如适用）
  - CLAUDE.md 配置文件
  - `.wise-config.json` state
- **与 `--no-plugin` 冲突**：两者同时设置时 `--no-plugin` 优先（并警告）。
- **自动检测**：环境中已设置 `WISE_PLUGIN_ROOT` 时自动启用 `--plugin-dir-mode`（除非 `--no-plugin` 覆盖）。

### `wise doctor --plugin-dir <path>`（新增）

**用法**：使用指定插件目录运行诊断。

```bash
wise doctor --plugin-dir /path/to/wise
wise doctor conflicts --plugin-dir /path/to/wise
```

- **作用**：将提供的路径解析为绝对路径，在 doctor 动作运行前设置 `WISE_PLUGIN_ROOT`，与 `launch.ts` 语义一致。
- **优先级**：显式 `--plugin-dir` flag 优先于已有 `WISE_PLUGIN_ROOT` env（不一致时警告）。
- **子命令支持**：适用于 `wise doctor` 与 `wise doctor conflicts`。
- **输出**：诊断结果反映你指定的插件目录。

### `WISE_PLUGIN_ROOT` 环境变量

**用法**：启动 Claude Code 时活跃插件根的权威来源。

```bash
export WISE_PLUGIN_ROOT=/path/to/wise
claude --plugin-dir /path/to/wise
```

- **设置方**：`wise --plugin-dir <path>` 启动器（经 `src/cli/launch.ts`）。
- **读取方**：HUD wrapper、setup 自动检测、doctor 诊断。
- **需要时**：直接使用 `claude --plugin-dir`（无 `wise` shim）时，下游组件须解析同一路径。
- **优先级**：显式 CLI flag 覆盖该 env（并警告）。

### 决策矩阵：使用哪个 flag/模式？

| 你的 setup                            | 启动命令                                                     | Setup 命令                       | 预期行为                                                            |
| ------------------------------------- | ------------------------------------------------------------ | -------------------------------- | ------------------------------------------------------------------- |
| **Marketplace 插件**（推荐）          | `wise` 或 `claude`（默认）                                   | `wise setup`                     | 常规：agents/skills 复制到 `~/.claude/`                             |
| **本地 dev checkout，使用 WISE shim** | `wise --plugin-dir /path`                                    | `wise setup --plugin-dir-mode`   | Dev 模式：从 `/path` 加载 agents/skills，不复制                     |
| **本地 dev checkout，无 WISE shim**   | `claude --plugin-dir /path` + `export WISE_PLUGIN_ROOT=/path` | `wise setup --plugin-dir-mode`   | Dev 模式 + 手动 env：从 `/path` 加载 agents/skills                  |
| **本地 dev，需要捆绑 skills**         | `wise --plugin-dir /path`                                    | `wise setup --no-plugin`         | 强制本地捆绑 skills 到 `~/.claude/skills/`，忽略插件                |
| **排查特定路径**                      | N/A                                                          | `wise doctor --plugin-dir /path` | 诊断显示 `/path` 的状态                                             |

### 运行时 status 语义

- **制品优先终端收敛**：team 监控器在存在 finalized state 制品时优先采用。
- **确定性解析失败处理**：畸形 result 制品视为终端 `failed`。
- **清理范围**：shutdown/cleanup 仅清除目标 team 的 `.wise/state/team/{teamName}`（不影响 sibling team）。

### 制品描述符与有界 handoff

WISE handoff 遵循制品优先纪律：

- **控制面**数据保持小而偏运维：队列 state、worker claim、会话 state 与互操作 task/message envelope。
- **数据面**制品保持持久：plans、prompts、specs、traces 与 result 文件。
- 大 payload 应通过描述符引用，而非复制进控制面 state。
- 当前低风险调用点显式遵循该拆分：
  - 共享互操作 state 将 oversized task description、task result 与 shared message 写入 `.wise/state/interop/artifacts/**`
  - prompt 持久化将持久 prompt/response 文件保留在 `.wise/prompts/**`，并通过 job status 记录暴露描述符元数据

规范描述符字段：

| 字段           | 含义                                                         |
| -------------- | ------------------------------------------------------------ |
| `kind`         | 制品类型，如 `plan`、`prompt`、`result` 或 `trace`           |
| `path`         | 持久制品路径                                                 |
| `contentHash?` | 可选完整性提示                                               |
| `createdAt`    | 制品创建时间戳                                               |
| `producer`     | 所属 worker/tool/skill                                       |
| `sizeBytes?`   | 可选大小，用于阈值检查                                       |
| `retention`    | 保留/所有权提示                                              |
| `expiresAt?`   | 短生命周期制品的可选过期时间                                 |

有界 handoff 策略：

1. 仅当调用点显式阈值允许时，小 payload 才内联。
2. 较大 payload 传递短摘要加描述符。
3. 将持久内容保留在 `.wise/plans/`、`.wise/prompts/` 等制品存储中，而非将完整正文嵌入队列或 status 记录。

## 目标工作流 UX：`/goal`、Ralph、Team、UltraQA、Ultragoal

WISE 提供多种完成目标型任务的方式。它们互补而非可互换。每个会话选择一个主循环权威，其他作为证据生产者或 handoff 目标。

| 表面                    | 运行时 owner                                      | 面向用户的承诺                                                | 完成证据                                                                     | 说明                                                                                                                                                            |
| ----------------------- | ------------------------------------------------- | ------------------------------------------------------------- | ---------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Claude Code `/goal`     | Claude Code 原生会话循环                          | 持续朝一个 stated 完成条件工作                                | `/goal` 评估器在对话中呈现的证据                                             | `/goal` 行为仅引用 Claude Code/Anthropic 文档或 changelog。不得将评估器描述为独立运行命令或读取文件。                                                           |
| Ralph                   | WISE 技能 + Stop-hook 强制                        | 单一 owner 持久实现直至 PRD story 通过                        | 测试/build/lint/typecheck 加 reviewer 验证                                   | 当正确性依赖 WISE 的 PRD、progress 与 reviewer 门控时优先。                                                                                                     |
| Team                    | WISE native/team 或 CLI team 运行时               | 在分配任务上协调多 agent 执行                                 | Worker 任务结果、commits、分阶段 `team-verify`/`team-fix` 证据               | 当 ownership 边界与并行 lane 重要时优先。                                                                                                                       |
| UltraQA                 | WISE QA 循环技能                                  | 重复 diagnose/fix 直至质量门通过                              | 每轮所请求 QA 目标的命令输出                                                 | 实现目标已知但验证仍失败时优先。                                                                                                                                |
| 仅制品 Ultragoal        | 持久目标 ledger/checkpoint/handoff 制品           | 跟踪目标 state 而不启动另一活跃循环                           | 目标制品、checkpoint、handoff prompt、附加命令/审查证据                      | 当 `/goal` 不可用、不安全或与活跃 WISE 循环冲突时优先。                                                                                                         |

### `/goal` 来源与证据边界

`/goal` 事实仅使用 Claude Code/Anthropic 来源，包括：

- Claude Code `/goal` 文档：<https://code.claude.com/docs/en/goal>
- Anthropic Claude Code changelog：<https://raw.githubusercontent.com/anthropics/claude-code/main/CHANGELOG.md>

不得以 OpenAI/Codex 文档作为 Claude Code `/goal` 的权威。在 WISE 文档、示例与 handoff prompt 中保持该限制显式：`/goal` 评估器判断可见对话证据，不应描述为独立执行 shell 命令、读取文件、检查隐藏 state，或替代 WISE 最终审查门控。

### 推荐冲突处理

当多个循环可能适用时，使用以下确定性策略：

1. **refuse**：若 Ralph、Team、UltraQA、autopilot 或其他 Stop-hook 循环已活跃，且新 `/goal` 会与续跑权威竞争。
2. **adopt_existing**：若 Claude Code `/goal` 已活跃，且 WISE 工作流可将证据附加到同一条件而不改变循环 ownership。
3. **artifact_only**：若因 hooks/信任/设置导致 `/goal` 不可用，或用户仅需持久规划、checkpoint 与证据捕获。

`/goal` 评估器成功可作为有用证据，但 WISE 完成仍须相关持久证明：命令输出、变更文件、reviewer 裁决、任务结果或发布制品。

更短的用户向选择指南见 [Mode Selection Guide](./shared/mode-selection-guide.md#goal-oriented-workflow-selection)。

<a id="claude-code-goal-adapter-design"></a>
## Claude Code `/goal` 适配器设计

WISE 将 Claude Code `/goal` 视为可 handoff 的原生执行循环，而非 WISE 完成的持久真相来源。设计契约文档见 [docs/design/CLAUDE_CODE_GOAL_ADAPTER.md](./design/CLAUDE_CODE_GOAL_ADAPTER.md)。

关键契约点：

- Claude Code `/goal` 事实仅可引用 Claude Code 或 Anthropic 来源。OpenAI/Codex 引用为对比来源，非 Claude Code 行为权威。
- 适配器渲染可测量的 `/goal <condition>` handoff；不得直接变更隐藏 Claude Code 会话 state。
- 确定性冲突策略恰为 `refuse`、`adopt_existing` 或 `artifact_only` 之一；竞争的 Ralph/autopilot/Stop-hook/Team/UltraQA 循环不得仅警告后继续。
- `/goal` 评估器成功是 WISE 最终审查的证据，而非完成本身；WISE 仍须可观察的命令/测试/文档证据。
- WISE 在 WISE 拥有的逻辑制品与 `.wise/` 解析路径下存储持久目标 ledger 与证据，而非硬编码 `.omx/` 路径。

### Code Simplifier Hook

`code-simplifier` Stop hook 在每次 Claude turn 后自动将最近修改的源文件委派给 `code-simplifier` agent。默认**禁用**，须在全局 WISE 配置文件中显式启用：

- Linux/Unix 默认：`${XDG_CONFIG_HOME:-~/.config}/wise/config.json`
- macOS/Windows legacy/默认路径：`~/.wise/config.json`
- 适用处仍读取 legacy `~/.wise/config.json` 作为回退。

**启用：**

```json
{
  "codeSimplifier": {
    "enabled": true
  }
}
```

**完整配置选项：**

```json
{
  "codeSimplifier": {
    "enabled": true,
    "extensions": [".ts", ".tsx", ".js", ".jsx", ".py", ".go", ".rs"],
    "maxFiles": 10
  }
}
```

| 选项         | 类型       | 默认值                                          | 说明                     |
| ------------ | ---------- | ----------------------------------------------- | ------------------------ |
| `enabled`    | `boolean`  | `false`                                         | opt-in 自动简化          |
| `extensions` | `string[]` | `[".ts",".tsx",".js",".jsx",".py",".go",".rs"]` | 考虑的文件扩展名         |
| `maxFiles`   | `number`   | `10`                                            | 每 turn 最多简化文件数   |

**工作原理：**

1. Claude stop 时，hook 运行 `git diff HEAD --name-only` 查找修改文件
2. 若发现修改的源文件，hook 注入消息要求 Claude 委派给 `code-simplifier` agent
3. agent 在不改变行为的前提下简化文件以提高清晰度与一致性
4. turn-scoped 标记防止 hook 在同一 turn 周期内多次触发

### 本地化触发（韩语 / 日语）

关键词检测器除上述英文触发短语外，还识别本地化别名。每个别名映射到与英文相同的技能/模式：

| 关键词           | 韩语        | 日语               |
| ---------------- | ----------- | ------------------ |
| `ralph`          | 랄프        | ラルフ             |
| `autopilot`      | 오토파일럿  | オートパイロット   |
| `ultrawork`      | 울트라워크  | ウルトラワーク     |
| `ralplan`        | 랄플랜      | ラルプラン         |
| `ultrathink`     | 울트라씽크  | ウルトラシンク     |
| `ccg`            | 씨씨지      | シーシージー       |
| `deep-interview` | 딥인터뷰    | ディープインタビュー |
| `tdd`            | 테스트 퍼스트 | テスト ファースト |
| `code-review`    | 코드 리뷰   | コード レビュー    |
| `security-review`| 보안 리뷰   | セキュリティ レビュー |
| `deepsearch`     | 딥 서치     | ディープ サーチ    |
| `analyze`        | 딥 분석     | ディープ アナライズ |

`cancelwise` / `stopwise` 无本地化别名（取消仅匹配英文 token）。

#### 本地化路由行为

- **子串匹配（激进路由）。** 韩语与日语无 ASCII 词边界，因此本地化别名以子串而非整词匹配。这是有意设计：嵌入更长名词短语中的别名仍会路由 — 例如 `コードレビュー記事を要約して`（「总结这篇 code-review 文章」）会激活 **code-review** 模式。若不想该行为，优先使用英文形式，或在 alias 周围组织措辞。
- **Reviewer 后缀守卫。** `code-review` / `security-review` 使用 negative lookahead，使「reviewer」类名词不触发审查模式：`(?!어)` 阻止韩语 리뷰어，`(?!ア)` 阻止任何日语 レビューア… 形式（如 レビューアー）。
- **信息性抑制。** 帮助式问题被抑制并直通，不激活模式 — 例如 alias 附近的韩语 `뭐야` / 日语 `とは` / `使い方`。
- **差异类问题。** 日语「差异」措辞 — `…の違いを教えて`/`違いを説明`/`違いを知りたい` 与 `どう違う`/`何が違う`/`どこが違う`（如 `ディープサーチと普通の検索の違いを教えて`）视为信息性并抑制。`違い` 后有工作动词（如 `違いを修正して`）**不**抑制，仍会激活。

### 示例

```bash
# In Claude Code:

# Maximum parallelism
ultrawork implement user authentication with OAuth

# Enhanced search

# Deep analysis

# Autonomous execution

# Parallel autonomous execution

# Persistence mode
ralph: refactor the authentication module

# Planning session

# TDD workflow
tdd: implement password validation

# Stop active orchestration

```

### HUD 配置（settings.json）

在 `~/.claude/settings.json` 中配置 HUD 元素：

```json
{
  "wiseHud": {
    "preset": "focused",
    "elements": {
      "cwd": true,
      "gitRepo": true,
      "gitBranch": true,
      "showTokens": true
    }
  }
}
```

| 元素               | 说明                                                                                                               | 默认值  |
| ------------------ | ------------------------------------------------------------------------------------------------------------------ | ------- |
| `cwd`              | 显示当前工作目录                                                                                                   | `false` |
| `gitRepo`          | 显示 git 仓库名                                                                                                    | `false` |
| `gitBranch`        | 显示当前 git 分支                                                                                                  | `false` |
| `wiseLabel`        | 显示 [WISE] 标签                                                                                                   | `true`  |
| `updateNotification` | WISE 标签后显示可用更新提示文本                                                                                  | `true`  |
| `contextBar`       | 显示上下文窗口使用量                                                                                               | `true`  |
| `agents`           | 显示活跃 agent 数量                                                                                                | `true`  |
| `todos`            | 显示 todo 进度                                                                                                     | `true`  |
| `ralph`            | 显示 ralph 循环状态                                                                                                | `true`  |
| `autopilot`        | 显示 autopilot 状态                                                                                                | `true`  |
| `showTokens`       | 显示 transcript 推导的 token 使用（`tok:i1.2k/o340`，可靠时含 `r...` reasoning 与 `s...` session total）           | `false` |

额外 `wiseHud` 布局与标签选项（顶层）：

| 选项       | 说明                                                                                | 默认值     |
| ---------- | ----------------------------------------------------------------------------------- | ---------- |
| `maxWidth` | HUD 行最大宽度（终端列数）                                                          | unset      |
| `wrapMode` | 设置 `maxWidth` 时为 `truncate`（省略号）或 `wrap`（在 `\|` 边界断行）            | `truncate` |
| `locale`   | HUD 标签预设。支持值：`en`、`zh-CN`                                                 | `en`       |
| `labels`   | 按标签 HUD 文本覆盖；仅支持已知 key                                                 | unset      |

`locale` 与 `labels` 仅影响 HUD 标签。英文仍为默认；不支持的 locale 值与未知 label key 会被忽略；显式 `labels` 覆盖 locale 预设。支持的 label key 为 `context`、`tokens`、`tool`、`agent`、`skill`、`ralph`、`background`、`thinking`、`staged`、`modified`、`untracked`、`ahead` 与 `behind`。

示例：

```json
{
  "wiseHud": {
    "locale": "zh-CN",
    "labels": {
      "context": "CTX"
    }
  }
}
```

可用预设：`minimal`、`focused`、`full`、`dense`、`analytics`、`opencode`

### 常见问题

| 问题                 | 解决方案                                                                         |
| -------------------- | -------------------------------------------------------------------------------- |
| 找不到命令           | 重新运行 `/wise:wise-setup`                                                      |
| Hooks 未执行         | 检查 hook 权限：`chmod +x ~/.claude/hooks/**/*.sh`                               |
| Agent 未委派         | 验证 CLAUDE.md 已加载：检查 `./.claude/CLAUDE.md` 或 `~/.claude/CLAUDE.md`       |
| LSP 工具不工作       | 安装语言服务器：`npm install -g typescript-language-server`                      |
| Token 限制错误       | 使用 `/wise:` 进行 token 高效执行                                                |

### 自动更新

Wise 包含静默自动更新系统，在后台检查更新。

特性：

- **限流**：最多每 24 小时检查一次
- **并发安全**：锁文件防止同时更新尝试
- **跨平台**：macOS 与 Linux 均可用

手动更新请重新运行插件安装命令，或使用 Claude Code 内置更新机制。

### 卸载

使用 Claude Code 插件管理：

```
/plugin uninstall wise@wise
```

或手动删除已安装文件：

```bash
rm ~/.claude/agents/{architect,document-specialist,explore,designer,writer,vision,critic,analyst,executor,qa-tester}.md
rm ~/.claude/commands/{analyze,autopilot,deepsearch,plan,review,ultrawork}.md
```
