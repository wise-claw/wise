# 架构

> wise 如何编排多智能体工作流。

## 概览

wise 让 Claude Code 能够通过基于技能的路由系统编排专业化智能体。它由四个相互衔接的系统构成：**Hooks** 检测生命周期事件，**Skills** 注入行为，**Agents** 执行专业化工作，**State** 在上下文重置后跟踪进度。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         WISE                                 │
│                     Intelligent Skill Activation                         │
└─────────────────────────────────────────────────────────────────────────┘

  User Input                      Skill Detection                 Execution
  ──────────                      ───────────────                 ─────────
       │                                │                              │
       ▼                                ▼                              ▼
┌─────────────┐              ┌──────────────────┐           ┌─────────────────┐
│  "ultrawork │              │   CLAUDE.md      │           │ SKILL ACTIVATED │
│   refactor  │─────────────▶│   Auto-Routing   │──────────▶│                 │
│   the API"  │              │                  │           │ ultrawork +     │
└─────────────┘              │ Task Type:       │           │ default +       │
                             │  - Implementation│           │ git-master      │
                             │  - Multi-file    │           │                 │
                             │  - Parallel OK   │           │ ┌─────────────┐ │
                             │                  │           │ │ Parallel    │ │
                             │ Skills:          │           │ │ agents      │ │
                             │  - ultrawork ✓   │           │ │ launched    │ │
                             │  - default ✓     │           │ └─────────────┘ │
                             │  - git-master ✓  │           │                 │
                             └──────────────────┘           │ ┌─────────────┐ │
                                                            │ │ Atomic      │ │
                                                            │ │ commits     │ │
                                                            │ └─────────────┘ │
                                                            └─────────────────┘
```

四个系统按顺序流转：

```
User Input --> Hooks (event detection) --> Skills (behavior injection)
           --> Agents (task execution) --> State (progress tracking)
```

## 智能体系统

### 概览

WISE 提供 19 个专业化智能体，分为 4 条工作流通道。每个智能体以 `wise:<agent-name>` 形式调用，并运行在相应的模型层级上。

### 构建/分析通道

覆盖从探索到验证的完整开发生命周期。

| 智能体 | 默认模型 | 职责 |
|-------|---------------|------|
| `explore` | haiku | 代码库发现、文件/符号映射 |
| `analyst` | opus | 需求分析、隐藏约束发现 |
| `planner` | opus | 任务排序、执行计划制定 |
| `architect` | opus | 系统设计、接口定义、权衡分析 |
| `debugger` | sonnet | 根因分析、构建错误排查 |
| `executor` | sonnet | 代码实现、重构 |
| `verifier` | sonnet | 完成度验证、测试充分性确认 |
| `tracer` | sonnet | 基于证据的因果追踪、竞争性假设分析 |

### 审查通道

交付前的质量关卡，用于发现正确性与安全问题。

| 智能体 | 默认模型 | 职责 |
|-------|---------------|------|
| `security-reviewer` | sonnet | 安全漏洞、信任边界、认证/授权审查 |
| `code-reviewer` | opus | 全面代码审查、API 契约、向后兼容性 |

### 领域通道

按需调用的领域专家。

| 智能体 | 默认模型 | 职责 |
|-------|---------------|------|
| `test-engineer` | sonnet | 测试策略、覆盖率、脆弱测试加固 |
| `designer` | sonnet | UI/UX 架构、交互设计 |
| `writer` | haiku | 文档、迁移说明 |
| `qa-tester` | sonnet | 通过 tmux 进行交互式 CLI/服务运行时验证 |
| `scientist` | sonnet | 数据分析、统计研究 |
| `git-master` | sonnet | Git 操作、提交、变基、历史管理 |
| `document-specialist` | sonnet | 外部文档、API/SDK 参考查阅 |
| `code-simplifier` | opus | 代码清晰度、简化、可维护性提升 |

### 协调通道

对其他智能体制定的计划与设计提出质疑；只有找不到缺口时，计划才算通过。

| 智能体 | 默认模型 | 职责 |
|-------|---------------|------|
| `critic` | opus | 计划与设计的缺口分析、多角度审查 |

### 模型路由

WISE 使用三个模型层级：

| 层级 | 模型 | 特点 | 成本 |
|------|-------|-----------------|------|
| LOW | haiku | 快速、低成本 | 低 |
| MEDIUM | sonnet | 性能与成本均衡 | 中 |
| HIGH | opus | 最高质量推理 | 高 |

按角色的默认分配：
- **haiku**：快速查找与简单任务（`explore`、`writer`）
- **sonnet**：代码实现、调试、测试（`executor`、`debugger`、`test-engineer`）
- **opus**：架构、战略分析、审查（`architect`、`planner`、`critic`、`code-reviewer`）

### 委派

工作通过 Task 工具委派，并配合智能模型路由：

```typescript
Task(
  subagent_type="wise:executor",
  model="sonnet",
  prompt="Implement feature..."
)
```

**应委派给智能体的情况：**
- 需要修改多个文件
- 需要重构
- 需要调试或根因分析
- 需要代码审查或安全审查
- 需要规划或调研

**可直接处理的情况：**
- 简单文件查找
- 直接回答问题
- 单条命令操作

### 智能体选择指南

| 任务类型 | 推荐智能体 | 模型 |
|-----------|-------------------|-------|
| 快速代码查找 | `explore` | haiku |
| 功能实现 | `executor` | sonnet |
| 复杂重构 | `executor` (model=opus) | opus |
| 简单 Bug 修复 | `debugger` | sonnet |
| 复杂调试 | `architect` | opus |
| UI 组件 | `designer` | sonnet |
| 文档编写 | `writer` | haiku |
| 测试策略 | `test-engineer` | sonnet |
| 安全审查 | `security-reviewer` | sonnet |
| 代码审查 | `code-reviewer` | opus |
| 数据分析 | `scientist` | sonnet |

### 典型智能体工作流

```
explore --> analyst --> planner --> critic --> executor --> verifier
(发现)    (分析)      (排序)      (审查)      (实现)       (确认)
```

### 智能体角色边界

| 智能体 | 负责 | 不负责 |
|-------|------|----------|
| `architect` | 代码分析、调试、验证 | 需求收集、规划 |
| `analyst` | 发现需求缺口 | 代码分析、规划 |
| `planner` | 制定任务计划 | 需求分析、计划审查 |
| `critic` | 审查计划质量 | 需求分析、代码分析 |

## 技能系统

### 概览

技能（Skills）是**行为注入**机制，用于改变编排器的运行方式。技能不会替换智能体，而是在现有智能体之上叠加能力。WISE 共提供 31 个技能（28 个用户可调用 + 3 个内部/流水线技能）。

### 技能分层

技能分三层组合：

```
┌─────────────────────────────────────────────────────────────┐
│  GUARANTEE LAYER (optional)                                  │
│  ralph: "Cannot stop until verified done"                   │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  ENHANCEMENT LAYER (0-N skills)                              │
│  ultrawork (parallel) | git-master (commits) | frontend-ui-ux│
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  EXECUTION LAYER (primary skill)                             │
│  default (build) | orchestrate (coordinate) | planner (plan) │
└─────────────────────────────────────────────────────────────┘
```

**公式：** `[执行技能] + [0-N 个增强技能] + [可选保障技能]`

示例：
```
Task: "ultrawork: refactor API with proper commits"
Active skills: ultrawork + default + git-master
```

### 如何调用技能

**斜杠命令：**
```bash
/wise:autopilot build me a todo app
/wise:ralph refactor the auth module
/wise:team 3:executor "implement fullstack app"
```

**魔法关键词** — 在自然语言中包含关键词即可自动激活对应技能：
```bash
autopilot build me a todo app      # 激活 autopilot
ralph: refactor the auth module    # 激活 ralph
ultrawork implement OAuth          # 激活 ultrawork
```

### 核心工作流技能

#### autopilot
从想法到可运行代码的完整自主五阶段流水线。
- 触发词：`autopilot`、`build me`、`I want a`
```bash
autopilot build me a REST API with authentication
```

#### ralph
重复循环，直到工作经核实完成才停止。`verifier` 智能体在循环退出前确认完成度。
- 触发词：`ralph`、`don't stop`、`must complete`
```bash
ralph: refactor the authentication module
```

#### ultrawork
最大并行度 — 同时启动多个智能体。
- 触发词：`ultrawork`、`ulw`
```bash
ultrawork implement user authentication with OAuth
```

#### team
协调 N 个 Claude 智能体，五阶段流水线：`plan → prd → exec → verify → fix`
```bash
/wise:team 3:executor "implement fullstack todo app"
```

#### ccg (Claude-Codex-Gemini)
同时分发给 Codex 与 Gemini；由 Claude 综合结果。
- 触发词：`ccg`、`claude-codex-gemini`
```bash
ccg: review this authentication implementation
```

#### ralplan
迭代式规划：Planner、Architect 与 Critic 循环直至达成共识。
- 触发词：`ralplan`
```bash
ralplan this feature
```

## Hooks

### 概览

Hooks 是响应 Claude Code 生命周期事件的代码。当用户提交提示、使用工具，或开始/结束会话时会自动运行。WISE 通过该 hook 系统实现智能体委派、关键词检测与状态持久化。

### 生命周期事件

Claude Code 提供 11 个生命周期事件。WISE 在这些事件上注册 hooks：

| 事件 | 触发时机 | WISE 用途 |
|-------|---------------|-----------|
| `UserPromptSubmit` | 用户提交提示 | 魔法关键词检测、技能注入 |
| `SessionStart` | 会话开始 | 初始设置、加载项目记忆 |
| `PreToolUse` | 使用工具前 | 权限校验、并行执行提示 |
| `PermissionRequest` | 请求权限时 | Bash 命令权限处理 |
| `PostToolUse` | 使用工具后 | 结果校验、更新项目记忆 |
| `PostToolUseFailure` | 工具失败后 | 错误恢复处理 |
| `SubagentStart` | 子智能体启动 | 智能体跟踪 |
| `SubagentStop` | 子智能体停止 | 智能体跟踪、输出验证 |
| `PreCompact` | 上下文压缩前 | 保留关键信息、保存项目记忆 |
| `Stop` | Claude 即将停止 | 持久模式强制、代码简化 |
| `SessionEnd` | 会话结束 | 会话数据清理 |

### system-reminder 注入

Hooks 通过 `<system-reminder>` 标签向 Claude 注入额外上下文：

```xml
<system-reminder>
hook success: Success
</system-reminder>
```

注入模式含义：

| 模式 | 含义 |
|---------|---------|
| `hook success: Success` | Hook 正常运行，按原计划继续 |
| `hook additional context: ...` | 附加上下文信息，请注意 |
| `[MAGIC KEYWORD: ...]` | 检测到魔法关键词，执行对应技能 |
| `The boulder never stops` | ralph/ultrawork 模式处于活跃状态 |

### 关键 Hooks

**keyword-detector** — 在 `UserPromptSubmit` 时触发。检测用户输入中的魔法关键词并激活对应技能。

**persistent-mode** — 在 `Stop` 时触发。当持久模式（ralph、ultrawork）活跃时，阻止 Claude 停止，直至工作经核实完成。

**pre-compact** — 在 `PreCompact` 时触发。在上下文窗口压缩前将关键信息保存到 notepad。

**subagent-tracker** — 在 `SubagentStart` 与 `SubagentStop` 时触发。跟踪当前运行的智能体；停止时验证输出。

**context-guard-stop** — 在 `Stop` 时触发。监控上下文用量，接近上限时发出警告。

**code-simplifier** — 在 `Stop` 时触发。默认禁用。启用后，Claude 停止时自动简化已修改文件。

通过配置启用：
```json
{
  "codeSimplifier": {
    "enabled": true,
    "extensions": [".ts", ".tsx", ".js", ".jsx", ".py", ".go", ".rs"],
    "maxFiles": 10
  }
}
```

### Hook 注册结构

WISE hooks 在 `hooks.json` 中声明。每个 hook 是带超时的 Node.js 脚本：

```json
{
  "UserPromptSubmit": [
    {
      "matcher": "*",
      "hooks": [
        {
          "type": "command",
          "command": "node scripts/keyword-detector.mjs",
          "timeout": 5
        }
      ]
    }
  ]
}
```

- `matcher`：hook 响应的模式（`*` 匹配所有输入）
- `timeout`：超时时间（秒）
- `type`：始终为 `"command"`（运行外部命令）

### 禁用 Hooks

禁用所有 hooks：
```bash
export DISABLE_WISE=1
```

跳过指定 hooks（逗号分隔）：
```bash
export WISE_SKIP_HOOKS="keyword-detector,persistent-mode"
```

## 状态管理

### 概览

WISE 将任务进度与项目知识存储在 `.wise/` 目录中。即使上下文压缩重置了上下文窗口，状态系统仍能保留关键信息。

### 目录结构

```
.wise/
├── state/                    # 按模式划分的状态文件
│   ├── autopilot-state.json  # autopilot 进度
│   ├── ralph-state.json      # ralph 循环状态
│   ├── team/                 # team 任务状态
│   ├── interop/              # 跨工具任务/消息信封
│   └── sessions/             # 按会话划分的状态
│       └── {sessionId}/
├── notepad.md                # 抗压缩备忘录
├── project-memory.json       # 项目知识库
├── plans/                    # 执行计划
├── notepads/                 # 按计划划分的知识捕获
│   └── {plan-name}/
│       ├── learnings.md
│       ├── decisions.md
│       ├── issues.md
│       └── problems.md
├── prompts/                  # 持久化的提示/响应产物
├── autopilot/                # autopilot 产物
│   └── spec.md
├── research/                 # 调研结果
└── logs/                     # 执行日志
```

### 控制面与数据面

WISE 将编排元数据与大型持久化产物分离：

- **控制面**：队列状态、工作者分配、会话状态，以及 `.wise/state/**` 下的跨工具任务/消息信封。
- **数据面**：计划、规格、提示、结果、追踪等持久化产物，位于 `.wise/plans/`、`.wise/notepads/`、`.wise/prompts/`、`.wise/state/interop/artifacts/**` 等路径。
- **具体交接示例**：
  - 共享 interop 状态将任务/消息元数据内联保存，而将过大的任务描述、任务结果与消息正文存于 `.wise/state/interop/artifacts/**`
  - 提示持久化将提示/响应文件存于 `.wise/prompts/**`，并在作业状态旁记录描述符元数据

**全局状态：**
- `~/.wise/state/{name}.json` — 用户偏好与全局配置

旧路径在读取时会自动迁移。

这种分离使调度器与状态检查保持轻量，同时让更丰富的产物可持续保存并便于检查。

### 产物描述符与有界交接

当交接需要引用大型产物时，优先使用描述符/句柄，而非将完整载荷内联粘贴。规范描述符结构如下：

| 字段 | 用途 |
|------|---------|
| `kind` | 产物类别（plan、prompt、result、trace 等） |
| `path` | 产物的持久化路径 |
| `contentHash?` | 可选的完整性/校验和提示 |
| `createdAt` | 创建时间戳 |
| `producer` | 所属工具、技能或工作者 |
| `sizeBytes?` | 可选载荷大小，用于阈值判断 |
| `retention` | 清理/归属的生命周期提示 |
| `expiresAt?` | 短期产物的可选过期时间 |

**有界交接规则：**

1. 当调用点的显式阈值允许时，小型载荷可内联保存。
2. 当载荷会膨胀控制面状态时，改用描述符 + 简短人类可读摘要。
3. 在描述符中保留归属/保留元数据，以便后续清理与审计可确定执行。

### Notepad

**文件：** `.wise/notepad.md`

Notepad 可经受上下文压缩。写入其中的内容在上下文窗口重置后仍然存在。

可使用 `notepad_write_manual` MCP 工具保存笔记，或使用 `notepad_write_priority` 保存持久笔记。

**MCP 工具：**

| 工具 | 说明 |
|------|-------------|
| `notepad_read` | 读取 notepad 内容 |
| `notepad_write_priority` | 写入高优先级备忘（永久保留） |
| `notepad_write_working` | 写入工作备忘 |
| `notepad_write_manual` | 写入手动备忘 |
| `notepad_prune` | 清理旧备忘 |
| `notepad_stats` | 查看 notepad 统计 |

**工作原理：**
1. 在 `PreCompact` 事件时，重要信息保存到 notepad
2. 压缩后，notepad 内容重新注入上下文
3. 智能体通过 notepad 恢复先前上下文

### 项目记忆

**文件：** `.wise/project-memory.json`

项目记忆是项目级知识的持久化存储，可跨会话保留。

**MCP 工具：**

| 工具 | 说明 |
|------|-------------|
| `project_memory_read` | 读取项目记忆 |
| `project_memory_write` | 覆盖整个项目记忆 |
| `project_memory_add_note` | 添加笔记 |
| `project_memory_add_directive` | 添加指令 |

**生命周期集成：**
- `SessionStart`：加载项目记忆并注入上下文
- `PostToolUse`：从工具结果中提取项目知识并保存
- `PreCompact`：在上下文压缩前保存项目记忆

### 会话作用域

**路径：** `.wise/state/sessions/{sessionId}/`

按会话隔离存储状态。同一项目上的多个会话可同时运行，互不冲突。

### 计划 Notepad（按计划的知识捕获）

**路径：** `.wise/notepads/{plan-name}/`

分别存储每个执行计划中的经验。

| 文件 | 内容 |
|------|----------|
| `learnings.md` | 发现的模式、成功做法 |
| `decisions.md` | 架构决策与理由 |
| `issues.md` | 问题与阻塞项 |
| `problems.md` | 技术债与注意事项 |

所有条目会自动打上时间戳。

### 集中式状态（可选）

默认情况下，状态保存在项目的 `.wise/` 目录中，worktree 删除时一并移除。

若要在删除 worktree 后仍保留状态，可设置 `WISE_STATE_DIR` 环境变量：

```bash
# 添加到 ~/.bashrc 或 ~/.zshrc
export WISE_STATE_DIR="$HOME/.claude/wise"
```

状态将保存在 `~/.claude/wise/{project-identifier}/`。项目标识符为 Git 远程 URL 的哈希，因此同一仓库在不同 worktree 间共享状态。

### 持久记忆标签

对关键信息，使用 `<remember>` 标签：

```xml
<!-- 保留 7 天 -->
<remember>API endpoint changed to /v2</remember>

<!-- 永久保留 -->
<remember priority>Never access production DB directly</remember>
```

| 标签 | 保留期限 |
|-----|-----------|
| `<remember>` | 7 天 |
| `<remember priority>` | 永久 |

## 验证协议

验证模块以证据确保工作完成：

**标准检查项：**
- BUILD：编译通过
- TEST：所有测试通过
- LINT：无 lint 错误
- FUNCTIONALITY：功能符合预期
- ARCHITECT：Opus 层级审查通过
- TODO：所有任务已完成
- ERROR_FREE：无未解决错误

证据必须新鲜（5 分钟内），并包含实际命令输出。
